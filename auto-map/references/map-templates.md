# Map Templates & Discovery Craft

The concrete craft behind `auto-map`: the platform target map, what each tier looks like when done well
(for both Claude's `CLAUDE.md` tree and Codex's `AGENTS.md` tree), the stamp format that makes
update-don't-clobber implementable, how to find the significant directories, and what to discover per
stack. Load in Phase 0–2.

## Platform target map (resolve BEFORE writing anything)

The platform contract (`claude|codex|dual`) fixes the WRITE TARGETS for the whole run. Writing anywhere
else is a contract violation — do it wrong and you poison the wrong agent or litter the repo with files
nothing reads.

| Tier | claude → writes | codex → writes | Notes |
|---|---|---|---|
| Tier 0 (always-on core) | root `CLAUDE.md` (or `.claude/CLAUDE.md`) | root `AGENTS.md` | ≤150 lines either way |
| Tier 1 (path-scoped cross-cutting rules) | `.claude/rules/<topic>.md` (`paths:` frontmatter) | **no Codex tier** — fold DOWN into the nearest nested `AGENTS.md` (or a linked Tier-3 doc if over budget) | Codex has no rules-glob loader; NEVER invent `.codex/rules` |
| Tier 2 (per-directory identity) | nested `<dir>/CLAUDE.md` | nested `<dir>/AGENTS.md` | ≤100 lines either way; same significant-directory list |
| Tier 3 (deep references) | linked `docs/*.md` in backticks | linked `docs/*.md` in backticks | identical — linked, never imported |

- **dual** = write BOTH the `CLAUDE.md` tree and the `AGENTS.md` tree from the SAME verified inventory
  (identical content, so the two never drift). The stamp records which platform each generated region
  serves.
- **Codex NEVER-DO list:** never write `CLAUDE.md`, never write `.claude/rules`, never invent a
  `.codex/rules` mirror, never touch private Codex memory (e.g. a user's own Codex config/history). Root
  + nested `AGENTS.md` are the ONLY Codex write targets.
- **Unsupported/unrecognized platform** → STOP before any write; do not default to `claude`.
- All of budgets, generation stamps, evidence paths, and `(verified:)`/`(unverified)` command markers
  apply to `AGENTS.md` output byte-for-byte the same as to `CLAUDE.md`. The format is identical; only the
  filename and the tier count (Codex has no Tier 1) differ.

## Generation stamps (load-bearing — the verifier checks them)

Every generated section is fenced by stamps; everything outside the fences is human-written and
PRESERVED verbatim on refresh:

```markdown
<!-- generated by auto-map v0.1.0 | 2026-07-06 | scope: root -->
...generated content...
<!-- /generated -->
```

Refreshing = replacing only the fenced regions. A file with no stamps and existing content is treated
as fully human-written: generated content is APPENDED in fences, never merged into their prose.

## Tier 0 — root `CLAUDE.md` (≤150 lines, loads every session)

```markdown
# <project>

<2-3 lines: what this is, for whom, the one architectural fact that changes decisions>

## Commands
<!-- generated by auto-map ... -->
- `pnpm dev` — local server on :3000 (verified: 2026-07-06)
- `pnpm -w exec vitest run` — unit suite (verified: 2026-07-06)
- `pnpm typecheck && pnpm lint` — the pre-commit gate (verified: 2026-07-06)
<!-- /generated -->

## Layout
<!-- generated by auto-map ... -->
- `src/api/` — HTTP surface (its `CLAUDE.md` has the handler map)
- `src/db/` — schema + migrations (never edit a shipped migration)
- `services/worker/` — queue consumer (deploys separately)
<!-- /generated -->

## Invariants
- Money is integer cents end-to-end; floats never touch amounts.
- All external input crosses `src/api/validate.ts` — no bare req.body reads.

Deep dives (read when needed): `docs/architecture.md`, `docs/deploy.md`
```

Rules: every command line carries `(verified: <date>)` or `(unverified)`. Layout lines are one per
top-level dir, each pointing at its nested map. Invariants are the ones that change decisions — not
style preferences (those are Tier 1 rules).

## Tier 1 — `.claude/rules/<topic>.md` (cross-cutting, path-scoped)

```markdown
---
paths:
  - "src/**/*.test.{ts,tsx}"
  - "tests/**"
---
# Testing conventions
- State assertions over interaction assertions; mock only at slow/nondeterministic boundaries.
- Every bug fix lands with its reproduction test in the same commit.
```

MUST have `paths:` frontmatter (the verifier flags unscoped rules — they load every session and
belong in the root instead). One topic per file; conventions that span folders, never folder identity.

## Tier 2 — nested `<dir>/CLAUDE.md` (≤100 lines, loads when Claude works there)

```markdown
<!-- generated by auto-map v0.1.0 | 2026-07-06 | scope: src/api -->
# src/api — HTTP surface

Express handlers; one file per resource. Registered in `src/api/index.ts` (a handler that isn't
exported there does not exist).

## Key files
- `index.ts` — router registry; ALL handlers register here
- `validate.ts` — the input boundary; every handler calls it first
- `middleware/auth.ts` — session → user resolution

## Local invariants
- Handlers never import from `src/db/` directly — repositories only (`src/repo/`).
- New routes need: handler + registry line + a case in `routes.test.ts`.

## Test this area
`pnpm exec vitest run src/api` (verified: 2026-07-06)
<!-- /generated -->
```

The four sections that earn their lines: what the folder IS, key files with roles, local invariants
(what a stranger would break first), and the area-scoped test command.

## Codex output — root `AGENTS.md` (Tier 0, ≤150 lines)

Same shape as the root `CLAUDE.md`, same budget, same stamp — only the filename differs. Because Codex
has no Tier-1 rules loader, cross-cutting conventions that would be `.claude/rules/*.md` on Claude are
either linked as Tier-3 docs or folded into the nested `AGENTS.md` where they apply.

```markdown
# <project>

<2-3 lines: what this is, for whom, the one architectural fact that changes decisions>

## Commands
<!-- generated by auto-map v0.1.0 | 2026-07-06 | scope: root | platform: codex -->
- `pnpm dev` — local server on :3000 (verified: 2026-07-06)
- `pnpm -w exec vitest run` — unit suite (verified: 2026-07-06)
<!-- /generated -->

## Layout
<!-- generated by auto-map v0.1.0 | 2026-07-06 | scope: root | platform: codex -->
- `src/api/` — HTTP surface (its `AGENTS.md` has the handler map)
- `src/db/` — schema + migrations (never edit a shipped migration)
<!-- /generated -->

## Conventions
- Money is integer cents end-to-end; floats never touch amounts. <!-- Tier-1 depth folds here on Codex -->

```

## Codex output — nested `<dir>/AGENTS.md` (Tier 2, ≤100 lines)

Identical to the nested `CLAUDE.md` template above with the filename swapped. Codex loads the `AGENTS.md`
nearest the file it is working on, so per-folder identity lives here — plus any path-scoped convention
that on Claude would have been a `.claude/rules` entry for this area.

```markdown
<!-- generated by auto-map v0.1.0 | 2026-07-06 | scope: src/api | platform: codex -->
# src/api — HTTP surface

Express handlers; one file per resource. Registered in `src/api/index.ts`.

## Key files
- `index.ts` — router registry; ALL handlers register here
- `validate.ts` — the input boundary; every handler calls it first

## Local invariants
- Handlers never import from `src/db/` directly — repositories only (`src/repo/`).

## Test this area
`pnpm exec vitest run src/api` (verified: 2026-07-06)
<!-- /generated -->
```

The `platform:` field in the stamp lets a `dual` run tell a Codex-generated region from a Claude one and
refresh each in place; a Claude run leaves it off (or writes `platform: claude`). The verifier only
requires the `<!-- generated by auto-map` prefix, so extra stamp fields are safe.

## Preserving human content BYTE-IDENTICALLY (both platforms)

Refresh replaces ONLY the bytes between a `<!-- generated by auto-map ... -->` and its matching
`<!-- /generated -->`. Everything outside those fences — human prose, ordering, whitespace, trailing
newlines — is copied through unchanged. Never reflow, re-indent, or reorder a human section to "tidy" it.
If a file has content but NO stamps, treat it as fully human-authored: append new generated material in a
fresh fenced block, never merge into the existing prose. If generated and human regions overlap or
ownership is otherwise ambiguous, STOP and surface it — do not rewrite on a guess.

## Finding the significant directories (Phase 1)

Mechanical signal first, judgment second:

```bash
# where does work actually happen? commit-churn by top-2-level dir, last 500 commits
git log -500 --format= --name-only | grep -v '^$' | cut -d/ -f1-2 | sort | uniq -c | sort -rn | head -20
```

Take the union of: high-churn dirs from above, module boundaries (a dir with its own package.json /
go.mod / Cargo.toml / __init__.py cluster), and public-surface dirs (routes, CLI entries, migrations,
infra). Then apply the rule of thumb: **if an engineer would pause to orient before editing there, it
gets a map**; a folder of three leaf files does not. In a monorepo every workspace package qualifies
automatically.

## Per-stack discovery requirements (Phase 1)

| Stack (detect by) | Must inventory |
|---|---|
| Next.js (`next` dep) | `app/`/`pages/` route tree, server actions, middleware, `next.config`, data-fetch layer |
| Node service (`package.json` scripts) | entrypoints, exported package surface (`exports` map), queues/consumers, env schema |
| Laravel (`artisan`) | `routes/*.php`, Models + migrations, Jobs/Queues, `config/*`, artisan commands |
| Rust (`Cargo.toml`) | workspace members, feature flags, `bin`/`lib` targets, unsafe hotspots, build.rs |
| Go (`go.mod`) | cmd/ entrypoints, internal/ boundaries, generated code (protobuf/sqlc), Makefile targets |
| Python (`pyproject`) | package entry points, CLI scripts, models/migrations (alembic/django), task runners |
| Monorepo (workspaces) | ALL of the above per package + the dependency edges between packages |

For every inventoried item record the EVIDENCE PATH it was read from — Phase 3's verifier and future
refreshes depend on claims being traceable.

## What does NOT go in the map

- Aspirations ("we plan to migrate to...") — map what IS.
- Style rules better enforced by a linter — point at the linter config instead.
- Anything already obvious from a filename — budget is spent on what a reader can't guess.
- Secrets, tokens, URLs with credentials — reference locations, never values.
